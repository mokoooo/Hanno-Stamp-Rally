1つ目は、ビーコンWebhookではLINEメッセージ送信だけ成功していて、stamps テーブルへの登録ができていないケースです。

まず、Codexには以下を渡してください。

現在、LINE Beaconは発火し、LINEメッセージの送信までは成功しています。
しかし、アプリ画面上のスタンプカードにスタンプが反映されません。

原因を以下の2系統に分けて改修してください。

1. Webhook側で `stamps` テーブルに正しくinsertできていない可能性
2. insertは成功しているが、フロント側のスタンプカードが再取得されていない可能性

# 現状確認

現在のスタンプカードは `GET /api/stamps/card` で取得しており、
`stampsTable` から現在ログイン中ユーザーのスタンプ取得状況を読み込んでいます。

そのため、LINE Beaconでスタンプを付与する場合も、最終的には必ず `stampsTable` に以下の行が入る必要があります。

- userId: usersTable.userId
- spotId: 1
- triggerType: "BEACON"

# 改修1: Beacon用のスタンプ付与処理を共通関数化する

`artifacts/api-server/src/routes/stamps.ts` または新規ファイルに、以下のようなサーバー内部用関数を作成してください。

目的:
QR用APIとBeacon用Webhookで、スタンプ登録ロジックを共通化する。

関数イメージ:

```ts
export async function applyStampBySpotId(params: {
  userId: string;
  spotId: number;
  triggerType: "QR" | "BEACON";
}) {
  const { userId, spotId, triggerType } = params;

  const spot = await db.query.spotsTable.findFirst({
    where: eq(spotsTable.id, spotId),
  });

  if (!spot) {
    return {
      status: "spot_not_found" as const,
      applied: false,
      spot: null,
    };
  }

  const existing = await db.query.stampsTable.findFirst({
    where: and(
      eq(stampsTable.userId, userId),
      eq(stampsTable.spotId, spotId),
    ),
  });

  if (existing) {
    return {
      status: "already_stamped" as const,
      applied: false,
      spot,
      stamp: existing,
    };
  }

  const [created] = await db.insert(stampsTable).values({
    userId,
    spotId,
    triggerType,
  }).returning();

  return {
    status: "applied" as const,
    applied: true,
    spot,
    stamp: created,
  };
}

既存の POST /api/stamps/apply も、この共通関数を使うようにリファクタしてください。
ただし、既存のQRコード機能を壊さないでください。

改修3: lineUserIdの不一致対策

現在のLIFFログインでは、場合によって line_uid:${profile.userId} の形式で保存される可能性があります。
一方、LINE Beacon Webhookの event.source.userId は通常 Uxxxxxxxx のようなLINE userIdです。

そのため、Webhook側のユーザー検索では以下の両方を試してください。

const user = await db.query.usersTable.findFirst({
  where: or(
    eq(usersTable.lineUserId, lineUserId),
    eq(usersTable.lineUserId, `line_uid:${lineUserId}`),
  ),
});

必要なimport:

import { eq, or } from "drizzle-orm";

これにより、LIFFログイン時の保存形式が Uxxxx でも line_uid:Uxxxx でも、Beaconイベントから同じユーザーを見つけられるようにしてください。

改修4: スタンプIDではなくspotId=1を明確に使う

今回「スタンプID:1」と表現しているが、現在のDB構造ではスタンプカードの1マス目は spotsTable.id = 1 に対応している可能性が高いです。

したがって、Beaconの処理では stampsTable.id = 1 ではなく、必ず以下をinsertしてください。

{
  userId: user.userId,
  spotId: 1,
  triggerType: "BEACON"
}

また、spotsTable.id = 1 が存在しない場合は、エラーにしてログに出してください。

改修5: フロント側のスタンプカードを自動更新する

LINE Beaconはサーバー側Webhookで処理されるため、アプリ画面を開いたままではReact Queryのキャッシュが古いままになる可能性があります。

artifacts/hanno-matsuri/src/pages/Home.tsx の useGetStampCard に、以下のいずれか、または両方を追加してください。

A. 一定間隔で再取得
const { data: stampCard, isLoading, error, refetch } = useGetStampCard(undefined, {
  query: {
    queryKey: getGetStampCardQueryKey(),
    refetchInterval: 10000,
    refetchOnWindowFocus: true,
    refetchOnReconnect: true,
  }
});
B. 画面復帰時に再取得
useEffect(() => {
  const handleVisibilityChange = () => {
    if (document.visibilityState === "visible") {
      refetch();
    }
  };

  document.addEventListener("visibilitychange", handleVisibilityChange);
  window.addEventListener("focus", refetch);

  return () => {
    document.removeEventListener("visibilitychange", handleVisibilityChange);
    window.removeEventListener("focus", refetch);
  };
}, [refetch]);

必要なimport:

import { useEffect } from "react";
